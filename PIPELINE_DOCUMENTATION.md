# Groupwise Multi-Turn Dataset Generation Pipeline

## Overview

This pipeline generates a multi-turn tool-calling dataset for an Android assistant using **Qwen3-8B** as the data generator. Tools are split into **11 thematic groups** (system, display, audio, etc.), and each group is generated by its own thin recipe script that delegates all heavy lifting to a shared core module.

The design follows one principle: **group scripts are thin recipes, the core is shared machinery.** Adding a new group is ~80 lines. Fixing a validation bug is one edit in the core, and all 11 groups benefit.

---

## End-to-End Flow (Single Group)

```
$ python gen_08_personal_info.py --tools-file tools.json --num-samples 20
                  │
                  ▼
   ┌──────────────────────────────────────────┐
   │ run_group(name, allowlist, seeds, out)   │
   └──────────────┬───────────────────────────┘
                  │
                  ├──── argparse → parse CLI flags
                  ├──── random.seed() / torch.manual_seed()    [reproducibility]
                  ├──── load_model_hf()  ── Qwen3-8B           [~3 min, ONCE]
                  │
                  ▼
   ┌──────────────────────────────────────────┐
   │ run_group_generation(...)                │
   └──────────────┬───────────────────────────┘
                  │
                  ├──── load_tools_filtered(tools.json, ALLOWLIST)
                  │         keeps only the 3 tools for this group
                  │
                  ├──── prepare_seeds(SEEDS, schema)
                  │         filter seeds + auto-coverage for missing tools
                  │
                  ├──── load existing JSONL → populate accepted_sigs
                  │         (resume support — re-running picks up where it left off)
                  │
                  ▼
   ╔══════════════════════════════════════════╗
   ║          MAIN LOOP                       ║
   ║   (until n_valid >= num_samples          ║
   ║    OR  attempts >= num_samples * 6)      ║
   ╚══════════════════════════════════════════╝
                  │
                  ├── next_seed()           pick scenario (round-robin)
                  ├── build_user_prompt()   intent + persona + hint
                  ├── build_chat_prompt()   wrap in Qwen3 chat template
                  │
                  ├── batch B=4 prompts together
                  │
                  ├── generate_batch()      ── LLM call (~30-40 s/batch)
                  │
                  │   for each generated string:
                  │     │
                  │     ├── strip <think>...</think> blocks
                  │     ├── extract_json()         parse model output
                  │     ├── validate_sample()      schema + target-tool coverage
                  │     ├── dedup check            (intent + tools_called signature)
                  │     ├── per-seed cap check     prevent over-sampling 1 seed
                  │     │
                  │     └── if all pass → append to group_NN_<name>.jsonl
                  │
                  └── print progress: [batch] valid=X/N attempts=Y sec=Z
                  
   On exit → close file, print final stats, count by seed
```

## File-by-File Breakdown

### 1. `run_all_groups.sh` — orchestrator

A small bash script that:

1. Takes 3 arguments — tools file, samples per group, model name.
2. Loops through `gen_01` → `gen_11` in order.
3. For each group, runs `python gen_NN_*.py --tools-file ... --num-samples N --model ...`.
4. Each script writes its own output file: `group_01_system_device.jsonl`, `group_02_display_ui.jsonl`, etc.
5. After all 11 finish, you can merge them: `cat group_*.jsonl > dataset_all_groups.jsonl`.

Usage:

```bash
bash run_all_groups.sh tool_suwon_ocfk_v2.json 20 Qwen/Qwen3-8B
```

**Behavior:** each group runs to its quota (or hits `max_attempts = num_samples * 6` and gives up early), then control passes to the next group. Per-group caps are enforced inside `generator_core` to prevent any single seed from dominating.

---

### 2. `gen_NN_<group>.py` — thin recipe

Every group script has exactly the same shape — only the data inside changes.

```python
from generator_core import run_group

GROUP_NAME = "personal_info"
ALLOWLIST  = {"calendar", "alarms", "accounts"}

SEEDS = [
    {
        "intent":            "What's on my calendar today?",
        "target_tools":      ["calendar"],
        "hint":              "Use calendar with today action.",
        "persona":           "casual",
        "expected_outcome":  "Today's events listed.",
    },
    # ... 14 more seeds
]

if __name__ == "__main__":
    run_group(GROUP_NAME, ALLOWLIST, SEEDS, "group_08_personal_info.jsonl")
```

The five seed fields:

- **intent** — what the user wants. Drives the scenario topic.
- **target_tools** — tools the assistant MUST call. Drives validation.
- **hint** — strategy guidance shown to the LLM in the prompt.
- **persona** — phrasing style (casual, professional, technical, conversational). Drives user-message variety.
- **expected_outcome** — used downstream by DeepEval to judge completeness.

That's it. No generation logic, no validation, no I/O — all of that lives in the core.

---

### 3. `generator_core.py` — the engine

This is where everything actually happens. Functions (in execution order):

#### `run_group(name, allowlist, seeds, output)`
CLI entry point. Parses args, seeds RNGs, loads the model **once**, then calls `run_group_generation()`. Loading the model outside the loop is critical — Qwen3-8B takes 2–5 min to load, and we'd waste hours if we reloaded it per sample.

#### `load_model_hf(model_id, load_in_4bit, ...)`
Loads Qwen3 via HuggingFace `AutoModelForCausalLM.from_pretrained`. Supports 4-bit quantization (`BitsAndBytesConfig`) for memory savings on the A6000s.

#### `load_tools_filtered(tools_file, allowlist)`
Reads the full Samsung tool catalog (39 tools), keeps only those whose name is in the group's allowlist, returns the filtered list + a schema index.

#### `prepare_seeds(seeds, schema)`
Validates seed `target_tools` against the schema, drops any seeds referencing tools not in the allowlist, and auto-generates a single-tool coverage seed for every tool that's in the allowlist but missing from `SEEDS`. Result: every tool in the group is guaranteed to appear in at least one prompt.

#### `next_seed(seeds, counts, cap)`
Round-robin picker that respects the per-seed cap. The cap is `max(3, 2 × ideal_share)` where `ideal_share = num_samples / num_seeds`. This prevents any one seed from producing 50% of the dataset.

#### `build_user_prompt(seed, persona)`
Composes the prompt the LLM actually sees: system prompt + intent + hint + persona + tool definitions.

#### `build_chat_prompt(...)`
Wraps the user prompt using the Qwen3 chat template (`<|im_start|>user ... <|im_end|>`).

#### `generate_batch(prompts, model, tokenizer, ...)`
Tokenizes B prompts together, calls `model.generate()` with batched attention, decodes the outputs. Batch size 4 is the sweet spot for a single A6000 with Qwen3-8B in 4-bit.

#### `extract_json(raw_output)`
Strips `<think>...</think>` blocks (Qwen3's reasoning), finds the JSON object in the remaining text, parses it. Returns `None` on failure.

#### `validate_sample(obj, schema, target_tools, strict)`
Checks five things:
1. The object has the required top-level fields (`turns`, `tools_called`, etc.).
2. Each turn has a valid role and content.
3. Every tool call references a real tool in the schema.
4. Every tool call's arguments match the tool's parameter schema.
5. The expected target tools all appear (unless `--no-strict-targets` is set).

#### Duplicate detection
A "signature" is built from the first user turn (normalized + truncated to 150 chars) plus a sorted list of tool names called. If the signature has been seen before, the sample is rejected. This catches obvious near-duplicates without being so strict that legitimate variation gets killed.

#### File writing
Successful samples are appended to `group_NN_<name>.jsonl` with `flush()` after every write. If you `Ctrl-C` and re-run, the script reads the existing file, rebuilds `accepted_sigs`, and continues from where it stopped.

## Output Format

Each line of `group_NN_<name>.jsonl` is one JSON object:

```json
{
  "id": "uuid-string",
  "group": "personal_info",
  "seed_intent": "What's on my calendar today?",
  "persona": "casual",
  "tools": [ {...tool schema...}, ... ],
  "turns": [
    {"role": "user", "content": "yo what's on my calendar today"},
    {"role": "assistant",
     "tools_called": [
       {"name": "calendar", "arguments": {"action": "today"}}
     ],
     "content": "You have 3 events today: ..."}
  ],
  "tools_called": [...],
  "expected_outcome": "Today's events listed.",
  "metadata": {...}
}
```

After running all 11 groups, merge with:

```bash
cat group_*.jsonl > dataset_all_groups.jsonl
```

## How to Run

### Run a single group

```bash
python gen_08_personal_info.py \
    --tools-file tool_suwon_ocfk_v2.json \
    --num-samples 50 \
    --model Qwen/Qwen3-8B \
    --batch-size 4 \
    --temperature 0.85 \
    --load-in-4bit
```

### Run all 11 groups in sequence

```bash
bash run_all_groups.sh tool_suwon_ocfk_v2.json 50 Qwen/Qwen3-8B
```

### Inspect progress

```bash
# Count samples per group
for f in group_*.jsonl; do
    echo "$f: $(wc -l < "$f") samples"
done
```

### Resume after interruption

Just re-run the same command. The script reads the existing JSONL, restores all signatures, and continues.

## Key CLI Flags

| Flag | Effect |
|------|--------|
| `--tools-file` | Path to the Samsung tool catalog JSON |
| `--num-samples` | Target sample count for the group |
| `--model` | HF model id or local path (default Qwen/Qwen3-8B) |
| `--batch-size` | Prompts per generate call (default 4) |
| `--temperature` | Sampling temperature (default 0.85; raise to 0.95 if mode-collapsing) |
| `--top-p` | Nucleus sampling cutoff |
| `--max-new-tokens` | Cap on generated length (default 2000) |
| `--load-in-4bit` | bitsandbytes 4-bit quantization (saves ~12GB VRAM) |
| `--no-strict-targets` | Skip the "must call all target tools" check |
| `--debug-dir` | Where to dump rejected raw outputs for inspection |
| `--seed` | RNG seed for reproducibility |

## How to Modify Things Confidently

| Change | Where to edit |
|--------|--------------|
| Add scenarios to a group | `SEEDS` list in `gen_NN_*.py` |
| Add a new group | New `gen_12_<name>.py` + add to `run_all_groups.sh` |
| Change output JSONL schema | The `record = {...}` dict in `generator_core.py` |
| Change validation rules | `validate_sample()` in `generator_core.py` |
| Change generator backend (vLLM, llama.cpp) | Only `load_model_hf()` + `generate_batch()` |
| Add an LLM-judge second pass | Insert between dedup check and write step |
| Make dedup more permissive | Tweak the signature in `_normalize` / `accepted_sigs` |

The two-tier design means you almost never have to touch both layers for the same change.
